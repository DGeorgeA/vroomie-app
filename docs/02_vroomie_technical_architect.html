Architecture Decision Records

Every significant technology choice is documented here with the context that drove the decision, the alternatives evaluated, and the consequences accepted. These are living records — status will evolve as the platform matures.

ADR-001 YAMNet over Custom CNN for Audio Classification ✓ Accepted

Decision

Use Google's pre-trained YAMNet model via TensorFlow.js (tf.loadGraphModel from TF Hub) for all audio classification.

Context & Rationale

YAMNet provides 1024-dimensional embeddings from AudioSet — 521 classes, pre-trained on 2M+ YouTube clips. It runs fully in-browser via TF.js GraphModel (~1.8MB), requires zero training infrastructure, and delivers production-quality embeddings from day one. A custom CNN would need labelled automotive data, GPU training cycles, and model hosting — too expensive for MVP.

Consequences

Locked to YAMNet's representation space — automotive sounds not in AudioSet may have weaker separation. Model URL is hardcoded (tfhub.dev/google/tfjs-model/yamnet/tfjs/1). No control over embedding quality for edge cases. Migration path: custom fine-tuned model in Phase 3.

ADR-002 ScriptProcessorNode over AudioWorklet for Capture ⚠ Deprecated

Decision

Use ScriptProcessorNode with 4096-sample buffer for real-time audio capture into a 16000-sample ring buffer (1-second windows).

Context & Rationale

ScriptProcessorNode is simpler: synchronous callback, no Worker/SharedArrayBuffer complexity, and sufficient for 1-second classification windows at 16kHz. AudioWorklet requires cross-origin isolation headers (COOP/COEP), a separate JS file, and message-passing — significant complexity for marginal gain when classification runs at ~1 Hz anyway.

Consequences

ScriptProcessorNode is deprecated per W3C spec. Audio processing runs on main thread, competing with React rendering. Chrome console shows deprecation warning. Priority migration target — see ADR-002 in roadmap (Phase 1).

ADR-003 OfflineAudioContext for WAV Decoding & Resampling ✓ Accepted

Decision

Decode reference WAV files using a temporary AudioContext.decodeAudioData(), then resample to exactly 16kHz via OfflineAudioContext.

Context & Rationale

YAMNet expects exactly 16kHz mono input. Reference WAVs may be recorded at 44.1kHz or 48kHz. OfflineAudioContext provides deterministic resampling — the browser's built-in SRC algorithm runs offline (not real-time), producing identical output across runs. Manual resampling (linear interpolation) introduces aliasing artefacts.

Consequences

Creates a temporary AudioContext per WAV file during fingerprint generation (6 files). Each context is properly closed after use. Adds ~200ms per file decode. Cross-browser SRC quality varies slightly but is acceptable for embedding comparison.

ADR-004 Cosine Similarity over Dynamic Time Warping ✓ Accepted

Decision

Use cosine similarity (threshold ≥ 0.75) to match live 1024-d embeddings against reference fingerprints. Reject DTW, Euclidean distance, and learned metrics.

Context & Rationale

Cosine similarity is O(n) for fixed-dimension vectors — computes in <0.1ms for 1024 floats. DTW is O(n²) and designed for variable-length sequences, not fixed embeddings. YAMNet's mean-pooled embeddings are unit-normalised in practice, making cosine the natural metric. Threshold of 0.75 accounts for real-world mic noise, reverb, and frequency response curves.

Consequences

Single-threshold binary decision — no learned decision boundary. Sensitivity fixed at 0.75 (some hardening at 0.82 in audioMatchingEngine for strict mode). False negatives on distant mic positions; false positives on ambient engine-like sounds. Mitigation: session accumulation requires consistent detection across multiple windows.

ADR-005 localStorage Cache over IndexedDB for Fingerprints ✓ Accepted

Decision

Cache generated YAMNet fingerprints in localStorage under key vroomie_yamnet_fingerprints_v6. Regenerate from Supabase WAVs on cache miss.

Context & Rationale

6 fingerprints × 1024 floats × 8 bytes/float ≈ 48KB + JSON metadata ≈ ~120KB total. Well within localStorage's 5–10MB limit. API is synchronous and simple (getItem/setItem). IndexedDB adds async complexity, cursor management, and transaction semantics — overkill for a flat array. idb package exists in deps but is unused for this path.

Consequences

No TTL — stale fingerprints persist until cache key version bumps (v6). No integrity validation — corrupted JSON silently re-parses. Device-bound — no cross-device sync. If localStorage is full, setItem throws and fingerprints regenerate each session (acceptable degradation).

ADR-006 Supabase over Firebase for Backend Platform ✓ Accepted

Decision

Use Supabase (PostgreSQL, Storage, Auth, RLS) as the sole backend platform. No Firebase, no custom API servers.

Context & Rationale

Supabase provides PostgreSQL (relational queries, JOINs, migrations), S3-compatible Storage (WAV/WebM hosting), GoTrue Auth, and Row-Level Security — all accessible via @supabase/supabase-js client SDK. Firebase's NoSQL (Firestore) makes relational queries awkward. Supabase's open-source model avoids vendor lock-in. PostgreSQL enables future analytics without ETL.

Consequences

Single point of dependency — Supabase outage = no persistence. Anon key exposed in client bundle (public, read-only bucket access only — writes protected by RLS). Connection string is environment-specific. Supabase free tier limits: 500MB DB, 1GB storage, 50K auth users.

ADR-007 Dual AudioContext Architecture ✓ Accepted

Decision

Run two concurrent AudioContexts: one at 16kHz (for ML inference via ScriptProcessor) and one at the device's native sample rate (for waveform visualisation via AnalyserNode).

Context & Rationale

YAMNet requires 16kHz input. Forcing the waveform visualiser to 16kHz produces jagged, low-fidelity canvas rendering. A native-rate context (typically 44.1kHz or 48kHz) provides smooth, high-resolution waveform data via analyser.getByteTimeDomainData() with fftSize=2048. Both contexts share the same MediaStream (mic) without interference.

Consequences

Two AudioContext instances on a single mic stream — some browsers limit to 6 concurrent contexts. Teardown sequence is critical: native context must close before extraction stops, which must happen after MediaRecorder's onstop fires. Memory overhead: ~2× audio graph allocation.

Concurrency Model

Vroomie runs entirely on the main thread today. Understanding the concurrency boundaries is critical for predicting latency under load and planning the Web Worker migration.

Main Thread (Synchronous)

  • React component rendering (AudioRecorder, AudioWaveform, full UI tree)
  • ScriptProcessorNode.onaudioprocess callback — copies samples into ring buffer
  • YAMNet model.predict() — TF.js inference (~50–150ms blocking)
  • Cosine similarity computation against all 6 reference fingerprints
  • Session anomaly accumulation and state updates

requestAnimationFrame Loop

  • AudioWaveform canvas rendering — single RAF loop, never re-mounts
  • State machine: idle → burst → live → stopping → idle
  • Idle state throttled to ~20fps via setTimeout(50ms)
  • Live/burst states run at full 60fps for smooth visualisation
  • All animation state stored in refs — zero React re-renders from animation

Async Boundaries (Promises)

  • tf.loadGraphModel() — model download and parsing (~2–5s, network-dependent)
  • fetch() for WAV reference files from Supabase Storage
  • OfflineAudioContext.startRendering() — offline WAV decode/resample
  • supabase.insert() and supabase.storage.upload() — report persistence
  • navigator.mediaDevices.getUserMedia() — mic permission prompt

Mutex & Event Ordering

  • isProcessing flag — prevents overlapping YAMNet calls in onaudioprocess
  • isExtracting module flag — guards re-entrant startExtraction()
  • isRecordingRef — React ref mirrors state for stable async closure reads
  • Critical sequence: MediaRecorder.stop() → onstop → handleAudioUpload() → stopExtraction()
  • If stream dies before onstop fires, MediaRecorder blob is empty — teardown order is strict

Technical Debt & Known Limitations

Honest assessment of architectural debt, deprecated APIs, and known gaps that technical architects should factor into adoption and planning decisions.

⚠️

ScriptProcessorNode is Deprecated Critical

The W3C Web Audio spec deprecated ScriptProcessorNode in favour of AudioWorkletNode. Chrome logs console warnings. Current implementation works but may be removed in future browser versions. Migration requires a separate .js worklet file, SharedArrayBuffer, and cross-origin isolation headers. Ref: audioFeatureExtractor.js:64

🧵

Single-Threaded ML Inference High

YAMNet model.predict() runs on the main thread, blocking React rendering and audio callbacks for 50–150ms per inference. Should be offloaded to a dedicated Web Worker with tf.setBackend('cpu') or 'wasm' for deterministic, non-blocking execution. Ref: mlEmbeddingEngine.js:69-74

🧪

No Automated Testing for Audio Pipeline High

Zero test coverage for audioFeatureExtractor, mlEmbeddingEngine, datasetLoader, and audioMatchingEngine. Vitest is installed but no audio pipeline tests exist. Ring buffer logic, RMS gating, cosine similarity, and fingerprint serialisation are all untested. Web Audio API requires mocking (OfflineAudioContext, AnalyserNode).

📊

Only 6 Reference Patterns Medium

The fingerprint library contains 6 WAV files covering alternator bearing, intake leak, misfire, motor starter, and piston faults. Production accuracy requires significantly more reference patterns per fault type, covering different RPMs, mic positions, engine sizes, and ambient conditions. Ref: datasetLoader.js:8-15

🔗

No Model Versioning Medium

YAMNet model URL is hardcoded as https://tfhub.dev/google/tfjs-model/yamnet/tfjs/1. No version pinning, no A/B testing of models, no rollback mechanism. If TF Hub changes the model weights at this URL, all clients get the new model silently. Ref: mlEmbeddingEngine.js:5

💾

localStorage Cache Has No TTL or Integrity Check Low

Fingerprints cached under key vroomie_yamnet_fingerprints_v6 persist indefinitely. No hash validation — corrupted JSON silently re-parses. Version is bumped manually in source code. No mechanism to force cache invalidation remotely. Ref: datasetLoader.js:5

Scalability Assessment

How each axis of scale affects the Vroomie architecture, considering its client-side ML nature and Supabase backend.

Horizontal Scale
N/A
Client-side ML — each device runs its own inference. No server compute to scale. User count is unlimited at the compute layer.
Vertical Scale
CPU/GPU Bound
Limited by device CPU for TF.js WASM backend, or GPU for WebGL. Low-end phones: ~300ms inference. Flagships: ~50ms.
Storage Scale
Independent
Supabase PostgreSQL and Storage scale independently of client count. 500MB free tier; paid tiers to 100GB+.
Model Scale
Replaceable
YAMNet can be swapped for larger/smaller models. Larger models increase download + inference latency proportionally.
Reference Patterns
O(n)
Adding more WAV references is O(n) in storage download and startup fingerprint generation. Cosine matching is O(n) per window.
Concurrent Sessions
1 per device
Single mic stream, single extraction pipeline. No multi-session support. One AudioContext pair per recording session.
📈 Scaling Inflection Point At ~50 reference patterns, fingerprint generation time (~200ms × n) will exceed 10 seconds. At this point, pre-computed fingerprints should be served as a JSON API from Supabase instead of generated client-side from WAV files.
